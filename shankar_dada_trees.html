All 15 NeetCode Tree Problems — Bubble Room Edition

The Bubble Tree

Every binary tree is a network of floating bubble rooms connected by tubes. Walk through it below — once you see it, you'll never forget it.

The Bubble Tree — What Everything Means

Bubble Room = Node
Each bubble is a floating room. When Circuit falls into a bubble, he looks at the wall. If there are vitals written on the wall (a number), it's a real room. If the wall is blank — no vitals — it's a dead-end.

In code: A bubble = node. Vitals on wall = node.val. No vitals = None.
Tubes = Left/Right Connections
Each bubble has at most 2 doors with tubes trailing out:
Lavender door + tube = Left (L for Lavender, L for Left)
Red door + tube = Right (R for Red, R for Right)

Each tube connects to a child bubble below.

In code: Lavender = node.left, Red = node.right.
Vitals = Node Value
The number written on the bubble wall. Could be 10, 5, -3, anything. Circuit reads it the moment he falls in.

No vitals = the bubble is empty, a dead-end. Circuit doesn't even enter — he climbs back out immediately.

In code: Vitals = node.val. No vitals = if not node: return ...
Leaf = Bubble with No Doors
A leaf bubble has vitals on the wall but no doors at all — no lavender door, no red door. It's the bottom of the tree. Circuit reads the vitals and climbs back out.

In code: not node.left and not node.right
Corridor = Path Between Two Bubbles (the tricky one!)
The key rule: There are NO direct connections between sibling bubbles. You can't go from Bubble 3 to Bubble 7 directly — they're separate floating rooms!
To get from one bubble to another, you must climb UP through the tube to their shared parent bubble, then slide DOWN the other tube. This U-shaped path through a parent is called a corridor.
Example: To go from Bubble 3 to Bubble 7: climb UP lavender tube to Bubble 5, then slide DOWN red tube to Bubble 7. That's 2 tubes = corridor length 2.

In code: left_depth + right_depth through a node = the corridor passing through that bubble. This is exactly what diameter and max path sum calculate!

The 4 Scenes

Each scene is a ritual Circuit (or Shankar) performs. Close your eyes, replay the scene, and the code writes itself.

Scene 1: Circuit Reports UP
Bottom-Up DFS — info flows UP from children to parent
THE RITUAL (same in every bubble)
❶ Circuit falls into a bubble. No vitals on the wall? Dead-end — tell above a default (0, True, None), climb back out.
❷ Opens lavender door, peeks through → gets an answer (left)
❸ Opens red door, peeks through → gets an answer (right)
❹ Combine left + right + my vitals into one answer.
❺ Tell above that answer.
No diary. The final answer is what the root bubble tells Shankar.
RITUAL → CODE
❶
Falls in. No vitals? Dead-end — tell above default, climb back out.
if not node: return 0
❷
Lavender door. Opens it, peeks through → gets an answer back.
L = dfs(node.left)
❸
Red door. Opens it, peeks through → gets another answer.
R = dfs(node.right)
❹❺
Combine and tell above. Two answers in his head. Combines them with his vitals. Climbs back out.
return 1 + max(L, R)
Code Anatomy — WHY this exact order?
Line 1: def dfs(node):
Line 2:     if not node: return BASE     ← FIRST! Falls in, no vitals → climb back out immediately
Line 3:     L = dfs(node.left)              ← Peeks through lavender door, gets answer
Line 4:     R = dfs(node.right)             ← Peeks through red door, gets answer
Line 5:     return COMBINE(L, R)            ← LAST! Has BOTH answers → combine and tell above
Base case is ALWAYS first — Circuit falls in, checks the wall FIRST. No vitals? Climb back out. If you put it after Line 3, you'd crash — None.left = AttributeError!
Peek through doors BEFORE combining — Circuit can't combine answers he doesn't have yet. If you return before peeking, you never visit any children!
Return is ALWAYS last — Bottom-up. Collect from below FIRST, then combine and tell above. Circuit can't climb out until he's done peeking.
Common Mistakes
WRONG — base case returns wrong value
if not node: return -1  # ???
if not node: return 1   # ???
RIGHT — think: what does an empty bubble report?
# Height: no vitals = 0 depth
if not node: return 0

# "Is it balanced?": empty = yes
if not node: return True
WRONG — doing work before peeking
def dfs(node):
    if not node: return 0
    result = 1 + ???  # combine WHAT?
    L = dfs(node.left) # too late!
RIGHT — peek first, THEN combine
def dfs(node):
    if not node: return 0
    L = dfs(node.left)   # peek
    R = dfs(node.right)  # peek
    return 1 + max(L, R) # THEN combine
Scene 2: Circuit + Diary
Bottom-Up DFS + Global Tracker — report goes UP, answer is in the diary
THE RITUAL
❶ Circuit falls into a bubble. No vitals on the wall? Dead-end — tell above 0, climb back out.
❷ Opens lavender door, peeks through → gets an answer (left)
❸ Opens red door, peeks through → gets an answer (right)
❹ Diary: combine left + right + something about me (corridor through me). Beat the diary? Overwrite.
❺ Tell above: me + the better side (above can only use one tube)
Answer is in the diary, NOT what root tells.
The diary vs the report — why two different numbers?
Standing in Bubble 5, Circuit has two things to do:

📓 Diary = L + R — "How long is the corridor passing THROUGH me?" Uses both tubes.
📤 Tell above = 1 + max(L, R) — "How deep can you reach through me?" Uses only the deeper side.

They're different because the bubble above connects to me through one tube. It can only extend the path in one direction. The corridor (both sides) is a separate measurement that goes in the diary.
RITUAL → CODE
⓪
Shankar hands Circuit the diary before sending him.
best = 0
❶
Falls in. No vitals? Dead-end. Climb back out. Don't write in diary.
if not node: return 0
❷❸
Lavender door. Red door. Peeks through both, gets two answers.
L = dfs(node.left)
R = dfs(node.right)
❹
Diary: corridor through this bubble = L + R. Better than diary? Overwrite.
best = max(best, L + R)
❺
Tell above: deeper single side + 1. Corridor stays in diary.
return 1 + max(L, R)
Scene 2 = Scene 1 + one extra moment. Between "peek through both doors" (❷❸) and "tell above" (❺), Circuit pauses to write in the diary (❹). Everything else is identical.
Common Mistakes
WRONG — returning corridor instead of depth

                   # use both your tubes
RIGHT — diary gets L+R, report gets max(L,R)

Scene 3: Lingam Mavayya's Badge
Top-Down DFS — constraints flow DOWN, tightening at each level
THE RITUAL
❶ Circuit falls into a bubble. Lingam shines his badge from above — Circuit's badge glows and updates with the constraint.
❷ No vitals on the wall? Dead-end — return True, climb back out.
❸ Check: do my vitals OBEY what's on the badge? No? Return False.
❹ Opens lavender door, glows the badge with updated values, peeks through → gets True/False
❺ Opens red door, glows the badge with updated values, peeks through → gets True/False
❻ Tell above: lavender AND red (both must obey)
Work FIRST, then peek. Lingam checks this bubble BEFORE going deeper. Opposite of Scene 1.
Badge vs Diary — completely different things
Badge (Scene 3) = constraint passed DOWN from above. Lingam shines it, Circuit's badge updates. It tightens at each level.
Diary (Scene 2) = global tracker written by Circuit. It tracks the best answer seen across ALL bubbles.

Badge = instructions IN. Diary = answers OUT. Never mix them.
RITUAL → CODE (Validate BST example)
❶
Falls in with badge. Lingam shines badge = (lo, hi) constraint.
def dfs(node, lo, hi):
❷
No vitals? No patient to violate anything. Return True, climb out.
if not node: return True
❸
Check badge. Do my vitals obey? No? Return False immediately.
if not (lo < val < hi): return False
❹❺
Glow badge with tighter values. Lavender: new ceiling = my vitals. Red: new floor = my vitals.
return dfs(L, lo, val)
   and dfs(R, val, hi)
Common Mistakes
WRONG — checking AFTER peeking through doors
left = dfs(node.left, lo, node.val)
right = dfs(node.right, node.val, hi)
if not (lo < node.val < hi):  # too late!
    return False
RIGHT — check badge BEFORE peeking deeper

    return False          # stop HERE!
return dfs(node.left, lo, node.val) \
   and dfs(node.right, node.val, hi)
WRONG — not tightening badge values
dfs(node.left, lo, hi)   # same badge!
dfs(node.right, lo, hi)  # never tightens!
RIGHT — glow badge with tighter values

dfs(node.left, lo, node.val)
# Red: new floor = my vitals
dfs(node.right, node.val, hi)
WRONG — forgot initial badge values

RIGHT — start with no restrictions

Scene 3 order: base case → CHECK badge → glow badge with tighter values → peek through doors. Check is BETWEEN base case and peeking.
Scene 4: Shankar Dada's Rounds
BFS — Shankar cracks bubbles, yanks tubes, loads the truck
THE RITUAL
❶ Root bubble falls onto the truck.
❷ Shankar tips the truck — all bubbles tumble onto the floor.
❸ He walks to each bubble on the floor one by one. Cracks it open, reads vitals, does the work. Yanks lavender tube — child bubble lands on the truck. Yanks red tube — child bubble lands on the truck.
❹ Floor empty. Level done. Record result.
❺ Tips the truck again. Repeats from ❸. Stops when he tips the truck and nothing falls out.
No Circuit. No recursion. Shankar does this himself. Truck = queue. Floor = current level. Children always land on the truck (next level), never on the floor (current level).
Why truck + floor = level separation
Floor = current level being processed.
Truck = next level accumulating.
They never mix. Children always land on the truck, never on the floor.
Tipping the truck = for _ in range(len(queue)) = start a new level.
RITUAL → CODE
❶
Root bubble falls onto the truck.
q = deque([root])
❷
Tips the truck. Count what tumbles onto the floor. That's this level.
while q:
  sz = len(q)
❸
Each bubble on the floor: crack open, read vitals. Yank lavender tube → child lands on truck. Yank red tube → same.
for _ in range(sz):
  n = q.popleft()
  level.append(n.val)
  if n.left: q.append(n.left)
  if n.right: q.append(n.right)
❹❺
Floor empty. Level done. Record result. Tips the truck again.

WRONG — no snapshot, levels mix

    node = q.popleft()   # as you add

RIGHT — tip the truck FIRST (snapshot)

for _ in range(sz):  # exactly this many
    node = q.popleft()
    ...

RIGHT — only yank if tube exists

1#226 Invert Binary TreeEasy

▶
Flowchart
Level-by-level? No. Info from above? No — swapping doesn't need badge. Answer at root? Yes. → Scene 1
The Ritual
❶ Falls in. No vitals? Return None — nothing to invert.
❷ Peeks through lavender door → gets the (already inverted) left subtree.
❸ Peeks through red door → gets the (already inverted) right subtree.
❹ Swap the two doors — lavender becomes red, red becomes lavender.
❺ Tell above: the swapped bubble.
What changes from the template?
Base case: return None (not 0 — we return nodes, not numbers).
Combine: Instead of math, we literally swap node.left, node.right = node.right, node.left.
Note: You can swap BEFORE or AFTER peeking — both work. Swapping first, then recursing into the (now-swapped) children is fine. 
def invertTree(root):
    if not root: return None
    root.left, root.right = root.right, root.left
    invertTree(root.left)
    invertTree(root.right)
    return root

Flowchart
Info from above? No. Answer at root? Yes — total depth. → Scene 1
The Ritual
❶ No vitals? Dead-end — tell above 0 (zero depth).
❷ Lavender door → gets depth of left side.
❸ Red door → gets depth of right side.
❹ Tell above: 1 + max(left, right). "Deeper side + 1 for me."
What changes from the template?
Base case: 0 (dead-end has zero depth).

def maxDepth(root):

3#110 Balanced Binary TreeEasy

▶
Flowchart
Info from above? No. Answer at root? Yes. → Scene 1
The Ritual
❶ No vitals? Tell above (True, 0) — empty is balanced with height 0.
❷❸ Peek both doors → each returns a PAIR: (is_balanced?, height).
❹ I'm balanced if: BOTH sides are balanced AND their heights differ by at most 1.
❺ Tell above: (my_balanced?, 1 + max(left_h, right_h))
What changes from the template?
Key insight: Circuit needs to report TWO things — balanced? AND height. So the return value is a tuple (bool, int), not just a number. This is common in Scene 1 when you need to carry multiple pieces of info upward. 
def isBalanced(root):
    def dfs(node):
        if not node: return (True, 0)
        lb, lh = dfs(node.left)
        rb, rh = dfs(node.right)
        balanced = lb and rb and abs(lh - rh) <= 1
        return (balanced, 1 + max(lh, rh))
    return dfs(root)[0]

4#100 Same TreeEasy

▶
Flowchart
→ Scene 1 — two Circuits in two bubble trees, comparing bubble by bubble.
The Ritual
❶ Both no vitals? Identical dead-ends → True.
❶ One has vitals, other doesn't? Structure differs → False.
❷❸ Peek lavender doors of both trees. Peek red doors of both trees.
❹ Same if: my vitals match AND left same AND right same.
What changes from the template?
Two inputs: dfs(p, q) instead of dfs(node) — Circuit walks two trees simultaneously.
Base case has 3 branches: both None → True, one None → False, both exist → check values and recurse. 
def isSameTree(p, q):
    if not p and not q: return True
    if not p or not q: return False
    return p.val == q.val and isSameTree(p.left, q.left) and isSameTree(p.right, q.right)

5#572 Subtree of Another TreeEasy

▶
Flowchart
→ Scene 1 — Circuit visits every bubble in the main tree, holding a “photo” of the subtree. At each bubble, he asks: "Does the tree starting HERE match the photo?"
The Ritual
❶ No vitals in main tree? Can't match → False.
❷ Check: does isSameTree(root, sub) match from here? Yes → True, done!
❸ No match? Peek lavender door OR red door — maybe the subtree is deeper inside.
What changes from the template?
Uses isSameTree as a helper. Two-layer recursion: outer recursion walks the main tree (isSubtree), inner recursion compares at each position (isSameTree).

def isSubtree(root, sub):
    if not root: return False
    if isSameTree(root, sub): return True
    return isSubtree(root.left, sub) or isSubtree(root.right, sub)

6#230 Kth Smallest in BSTMedium

▶
Flowchart
→ Scene 1 (in-order variant)
The Ritual
Key BST trick: In a BST, if Circuit always goes through the lavender door first, then reads vitals, then goes through the red door — he visits bubbles in sorted order (smallest to largest). This is called in-order traversal.

So: just do in-order traversal, count down from K. When count hits 0, that's the Kth smallest.
What changes from the template?
Uses a stack instead of recursion — same idea, but iterative so we can stop early at the Kth element without visiting the entire tree.
Inner while loop: keeps going left (lavender) until dead-end. Then pops, reads vitals (that's the next smallest), goes right (red). 
def kthSmallest(root, k):
    stk, n = [], root
    while stk or n:
        while n:                 # keep going lavender (left)
            stk.append(n)
            n = n.left
        n = stk.pop()           # pop = next smallest
        k -= 1
        if k == 0: return n.val  # found Kth!
        n = n.right              # now try red (right)

7#105 Construct from Preorder+InorderMedium

▶
Flowchart
→ Scene 1 (build mode — Circuit builds bubbles instead of visiting them)
The Ritual
Key insight:
• Preorder = the order Circuit first enters each bubble (root first). So preorder[0] is always the root.
• Inorder = everything LEFT of root in inorder is the left subtree, everything RIGHT is the right subtree.

❶ Take the next value from preorder → that's the current bubble's vitals.
❷ Find that value in inorder → everything to its left = lavender subtree, everything to its right = red subtree.
❸ Recursively build lavender side, then red side.
❹ Return the constructed bubble.
What changes from the template?
Base case: lo > hi → no elements left → return None.
Return: a newly created TreeNode (building, not traversing).
Index map: idx = {v:i for i,v in enumerate(inorder)} lets you instantly find where a value sits in inorder. 
def buildTree(pre, ino):
    idx = {v: i for i, v in enumerate(ino)}
    pi = [0]                         # pointer into preorder
    def build(lo, hi):
        if lo > hi: return None
        v = pre[pi[0]]; pi[0] += 1   # next root from preorder
        n = TreeNode(v)
        m = idx[v]                    # find split in inorder
        n.left = build(lo, m - 1)     # left of split = lavender
        n.right = build(m + 1, hi)    # right of split = red
        return n
    return build(0, len(ino) - 1)

8#297 Serialize / DeserializeHard

▶
Flowchart
→ Scene 1 — two separate Scene 1 passes: one to serialize (read tree → string), one to deserialize (string → build tree).
Serialize (tree → string)
Circuit does a preorder walk. At each bubble:
❶ No vitals? Append "N" (marks dead-end) and return.
❷ Has vitals? Append the value, then peek lavender, then red.
Join everything with commas → "10,5,3,N,N,7,N,N,15,N,20,N,N"
Deserialize (string → tree)
Read tokens one by one from the string:
❶ Token is "N"? Return None (dead-end).
❷ Token is a number? Create a bubble with those vitals. Recursively build lavender child, then red child.
The preorder ensures tokens come in the right order. 
class Codec:
    def serialize(self, root):
        r = []
        def dfs(n):
            if not n:
                r.append("N"); return
            r.append(str(n.val))
            dfs(n.left); dfs(n.right)
        dfs(root)
        return ",".join(r)

    def deserialize(self, data):
        it = iter(data.split(","))
        def build():
            v = next(it)
            if v == "N": return None
            n = TreeNode(int(v))
            n.left = build()
            n.right = build()
            return n
        return build()

9#543 DiameterEasy

▶
Flowchart
Info from above? No. Answer at root? NO — the longest path could pass through ANY bubble, not necessarily the root. → Scene 2
The Ritual
❶ No vitals? Dead-end — tell above 0.
❷ Lavender door → gets depth of left side.
❸ Red door → gets depth of right side.
❹ Diary: left + right = path through me using both tubes. Beat diary? Overwrite.
❺ Tell above: 1 + max(left, right) = my depth (one tube only — above can only reach me through one tube).
What changes from the template?
Diary: left + right (both sides combined = path through this bubble).
Tell above: 1 + max(left, right) (same as max depth! above can only use one arm).
Diary starts at 0 — diameter is always ≥ 0 (it counts edges, not values).
Final answer: return diary, NOT the root's return value. 
def diameterOfBinaryTree(root):
    diary = 0
    def dfs(node):
        nonlocal diary
        if not node: return 0
        left = dfs(node.left)
        right = dfs(node.right)
        diary = max(diary, left + right)   # path through me
        return 1 + max(left, right)        # my depth
    dfs(root)
    return diary

10#124 Max Path SumHard

▶
Flowchart
"Does NOT need to pass through root" → answer at ANY bubble → Scene 2. Exact same pattern as diameter, but with vitals sum instead of edge count.
The Ritual
❶ No vitals? Dead-end — tell above 0.
❷❸ Peek both doors → get best sum from each side. If negative, treat as 0 (don't use that tube — a negative arm makes the sum worse).
❹ Diary: max(left,0) + my vitals + max(right,0) = best path through me using both tubes.
❺ Tell above: my vitals + max(left, right, 0) = best straight line through me (one tube only).
What changes from diameter?
Uses vitals (node.val) in the calculation, not just edge count.
max(left, 0): If a side returns negative, pretend it's a dead-end. A negative arm makes the path worse.
Diary starts at -infinity — values can be negative! If all vitals are -5, the answer is -5, not 0.
Tell above uses val: node.val + max(l, r) instead of 1 + max(l, r).
Tell above vs Diary (Max Path Sum):
Tell above = val + max(L, R, 0) — straight line, one direction. Above can only reach me through one tube.
Diary = max(L,0) + val + max(R,0) — U-shaped path through me using both tubes.
def maxPathSum(root):
    diary = float('-inf')
    def dfs(n):
        nonlocal diary
        if not n: return 0
        l = max(0, dfs(n.left))               # ignore negative
        r = max(0, dfs(n.right))              # ignore negative
        diary = max(diary, l + n.val + r)     # path through me
        return n.val + max(l, r)              # straight line up
    dfs(root)
    return diary

def isValidBST(root):
    def dfs(n, lo, hi):
        if not n: return True
        if not (lo < n.val < hi): return False
        return dfs(n.left, lo, n.val) and dfs(n.right, n.val, hi)
    return dfs(root, float('-inf'), float('inf'))

def goodNodes(root):
    def dfs(n, mx):
        if not n: return 0
        good = 1 if n.val >= mx else 0
        mx = max(mx, n.val)
        return good + dfs(n.left, mx) + dfs(n.right, mx)
    return dfs(root, root.val)

13#235 LCA of BSTMedium

▶
Flowchart
Info from above? YES — BST ordering tells Circuit which tube to take. → Scene 3 (simplified — see below)
The Ritual
Circuit falls into the root bubble with two target values (p and q) in mind.

❶ Read my vitals. Compare p and q against them.
❷ Both targets < my vitals? Both are on the lavender (left) side → go through lavender tube.
❸ Both targets > my vitals? Both are on the red (right) side → go through red tube.
❹ They split? One is left, one is right (or one equals me) → I'm the meeting point! Return me.
Why is this iterative (while loop) instead of recursive?
Scene 3 shortcut: In normal Scene 3, Circuit peeks through BOTH doors. But in a BST, Circuit only ever goes through ONE tube at each step — either lavender or red, never both. So there's no branching, just a straight walk down. A while loop is simpler than recursion when you only go one direction.

Think of it this way: Circuit doesn't need to "peek and come back" — he just keeps walking down the single correct path until p and q split apart. The moment they split, he's found the LCA. 
def lowestCommonAncestor(root, p, q):
    node = root
    while node:
        if p.val < node.val and q.val < node.val:
            node = node.left          # both on lavender side
        elif p.val > node.val and q.val > node.val:
            node = node.right         # both on red side
        else:
            return node               # they split here = LCA!

Flowchart
"Return values level by level" → Scene 4 Shankar's Truck!
The Ritual
❶ Root bubble falls onto the truck.
❷ Shankar tips the truck — all bubbles tumble onto the floor. Counts them: sz = len(q).
❸ Walks to each bubble on the floor one by one. Cracks it open, reads vitals, records it. Yanks lavender tube → child lands on truck. Yanks red tube → child lands on truck.
❹ Floor empty = level done. Records this level's vitals as a list.
❺ Tips the truck again. Repeats until nothing falls out.
What changes from the template?
This IS the template. Level order is the purest Scene 4 problem. Every other BFS problem is a tweak of this.
Result: [[10], [5, 15], [3, 7, 20]] — a list of lists, one per level. 
from collections import deque
def levelOrder(root):
    if not root: return []
    res, q = [], deque([root])
    while q:                              # truck not empty?
        lvl = []
        for _ in range(len(q)):           # tip truck → process floor
            n = q.popleft()              # crack open
            lvl.append(n.val)            # read vitals
            if n.left: q.append(n.left)   # yank lavender → truck
            if n.right: q.append(n.right) # yank red → truck
        res.append(lvl)                  # level done!
    return res

def rightSideView(root):
    if not root: return []
    res, q = [], deque([root])
    while q:
        for i in range(sz := len(q)):
            n = q.popleft()
            if i == sz - 1:              # last on this floor!
                res.append(n.val)
            if n.left: q.append(n.left)
            if n.right: q.append(n.right)
    return res
